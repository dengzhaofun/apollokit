> This page is for engineering teams. For product background and use cases, see [Product Overview](./index).

ApolloKit abstracts "business action → event" into a unified event hub. Every module, after a key operation (check-in success, exchange completion, mail read, lottery draw…), emits a normalized event into the internal archive and, optionally, pushes it to your downstream via Webhook.

## Event model

Events come in two kinds:

- **Internal**: produced by ApolloKit modules themselves. Schema is owned by ApolloKit and kept backward-compatible.
- **External**: custom events you report into the hub. Schema is owned by you. External events can be promoted to **canonical** (takes over the name and freezes fields) to prevent schema drift over time.

Event names use dot-separated namespaces, e.g.:

```
check_in.checked_in
exchange.completed
item.granted
mail.claimed
task.progress
lottery.drawn
```

The full event list evolves with `logger.record(...)` calls in `apps/server/src/modules/*/service.ts`. The "Event Catalog" page in the dashboard shows all events the current org has reported + inferred schema.

## Browse the event catalog

Admin API:

```http
GET /api/event-catalog
x-api-key: <admin-api-key>
```

Returns the merged view (internal + external). Each entry includes the event name, current status (`inferred` / `canonical`), field schema, and most-recent occurrence.

```http
GET /api/event-catalog/{name}
PATCH /api/event-catalog/{name}   # External only; auto-promotes to canonical
```

## Subscribing to webhooks

> In MVP, webhook subscription + retry delivery is shipped for the activity module (`/api/activity/webhooks/*`); generalization to the global event hub's subscription list is on the roadmap. This page describes **the contract** — what payload shape you get once subscriptions are enabled.

Flow:

1. Register an endpoint (an HTTPS URL of yours accepting POST) for an event name (or prefix match) via dashboard or API.
2. ApolloKit issues a webhook secret for that subscription.
3. When the event fires, ApolloKit asynchronously delivers a signed JSON POST to your endpoint.

## Payload shape

```json
{
  "id": "evt_01hx...",
  "name": "check_in.checked_in",
  "organizationId": "org_...",
  "endUserId": "player_42",
  "occurredAt": "2026-04-21T03:14:15.123Z",
  "data": {
    "configId": "...",
    "configAlias": "daily",
    "currentStreak": 3,
    "justCompleted": false
  },
  "delivery": {
    "attempt": 1,
    "webhookId": "wh_..."
  }
}
```

Field contract:

| Field | Type | Meaning |
|---|---|---|
| `id` | string | Unique event id — the **idempotency key**. Retries reuse the same id. |
| `name` | string | Event name. |
| `organizationId` | string | Owning project id. |
| `endUserId` | string \| null | Player that triggered the event (null for system events). |
| `occurredAt` | ISO8601 | When the event actually occurred (not when it was delivered). |
| `data` | object | Event-specific fields. Schema at `/api/event-catalog/{name}`. |
| `delivery.attempt` | number | Delivery attempt count, starting at 1. |
| `delivery.webhookId` | string | Subscription id, for reconciliation. |

## Signature verification

Each delivery carries two headers:

```
x-apollokit-signature: sha256=<hmac-hex>
x-apollokit-timestamp: <unix-seconds>
```

Recommended verification algorithm:

```ts
const payload = `${timestamp}.${rawBodyText}`;
const expected = crypto.createHmac('sha256', webhookSecret)
  .update(payload)
  .digest('hex');
// constant-time comparison
```

Checks:

1. Compare `sha256=...` against your computed hex.
2. Verify `timestamp` is within 5 minutes of server time (replay defense).
3. Only then process `data`.

## Idempotent handling

Because retries exist, the same `id` may reach you 2+ times. Dedupe on the receiver:

```ts
const seen = await kv.get(`webhook:${event.id}`);
if (seen) return new Response('', { status: 200 });
await kv.put(`webhook:${event.id}`, '1', { expirationTtl: 24 * 3600 });
// ...handle...
```

## Retry & backoff

Current contract: 2xx = success; 3xx/4xx/5xx or timeout (5s) = failure. Failures retry with exponential backoff, up to 8 attempts spread across 24 hours. After that, events move to `dead_letter`, visible in the dashboard.

## Local development

Use `webhook.site` or `smee.io` for a throwaway endpoint to inspect real payloads first, then implement signature verification. Before production, always turn on signature verification and disable dev subscriptions.
