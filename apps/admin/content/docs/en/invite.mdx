The invite module implements "veterans bring rookies for rewards": each player has a dedicated invite code; new player redeems on signup, binding the invite relationship; on reaching conditions (payment, leveling up, etc.) the `qualified_at` flag is set, and the inviter claims the reward.

## Data model

- **invite_settings** (one row per org): reward config, qualifier conditions, bilateral-existence requirement, code-format rules.
- **invite_codes**: per-player invite code (rotatable), unique on `(orgId, code)`.
- **invite_relationships**: inviter → invitee binding, with `bound_at` / `qualified_at` / `reward_claimed_at`.

## Admin API

```http
PUT /api/invite/settings
GET /api/invite/settings
GET /api/invite/relationships              view bindings and qualification states
```

## Client API

```http
GET  /api/invite/end-user/my-code
POST /api/invite/end-user/rotate-code       rotate to a new code
POST /api/invite/end-user/redeem           body: { code }  new player redeems
POST /api/invite/end-user/mark-qualified   business-side flags qualification
POST /api/invite/end-user/claim-reward     inviter claims the reward

GET  /api/invite/end-user/my-invitees
```

## Business rules

- **One-shot binding**: an invitee can only bind one inviter. `redeem` is idempotent — repeating with the same code returns "already bound".
- **Self-invite defense**: players can't redeem their own code.
- **Qualifier threshold**: `qualified_at` is set by business code when conditions are met (or via event triggers). Without it, the inviter cannot claim.
- **Reward idempotency**: `claim-reward` can only be claimed once per relationship; the service uses atomic `UPDATE ... WHERE reward_claimed_at IS NULL RETURNING` to prevent double claims under concurrency.
- **Code rotation**: rotating invalidates the old code immediately but retains already-bound relationships.

## Combining with other modules

- [friend](./friend): auto-add as friend on successful invite (optional).
- [item](./item) / [currency](./currency): rewards flow through these.
- [task](./task): turn "invite N qualified users" into a task.
- [mail](./mail): notify the inviter via mail when qualification is reached.

## Events

`invite.bound` / `invite.qualified` / `invite.reward_claimed`.
