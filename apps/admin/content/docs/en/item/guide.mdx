> This page is for engineering teams. For product background and use cases, see the [Product Overview](./index).

The item module is the platform's base resource layer, managing virtual-item definitions, categories and per-user holdings. Check-in rewards and the exchange system flow through it for actual resource movements.

## Core concepts

### ItemCategory

Logical grouping of items — "currency", "consumables", "materials":

| Field | Type | Meaning |
|------|------|---------|
| `name` | string | Category name |
| `alias` | string? | Org-unique alias |
| `icon` | string? | Icon URL |
| `sortOrder` | number | Sort weight |
| `isActive` | boolean | Enable flag |

### ItemDefinition

Defines an item's properties and stacking rules:

| Field | Type | Meaning |
|------|------|---------|
| `name` | string | Item name |
| `alias` | string? | Org-unique alias (e.g. `gold`, `gem`) |
| `categoryId` | UUID? | Category |
| `description` | string? | Description |
| `icon` | string? | Icon URL |
| `stackable` | boolean | Stackable flag (default `true`) |
| `stackLimit` | number? | Per-stack cap (`null` = unlimited) |
| `holdLimit` | number? | Per-user hold cap (`1` = unique item) |
| `isActive` | boolean | Enable flag |

### Stacking rules

Inventory storage depends on stacking settings:

| Scenario | Behavior |
|----------|----------|
| `stackable=true`, `stackLimit=null` | Unlimited stacking, one row per (user, item) — ideal for currencies |
| `stackable=true`, `stackLimit=99` | Bounded stacking, auto-splits at 99 |
| `stackable=false` | Not stackable, each instance is its own row (quantity always 1) |

### InventoryView

Aggregate view when querying inventory:

```json
{
  "definitionId": "uuid",
  "definitionAlias": "gold",
  "definitionName": "Gold",
  "icon": "https://...",
  "stackable": true,
  "totalQuantity": 150,
  "stacks": [
    { "id": "stack-uuid-1", "quantity": 99, "instanceData": null },
    { "id": "stack-uuid-2", "quantity": 51, "instanceData": null }
  ]
}
```

### ItemGrantLog

Every grant/deduct appends an audit entry:

| Field | Meaning |
|------|---------|
| `delta` | Change (positive=grant, negative=deduct) |
| `source` | Source identifier (e.g. `check_in`, `exchange`, `admin_grant`) |
| `sourceId` | Idempotency key |
| `quantityBefore` | Quantity before the change |
| `quantityAfter` | Quantity after the change |

## Admin API

### Category CRUD

```http
POST   /api/item/categories
GET    /api/item/categories
GET    /api/item/categories/{key}        # UUID or alias
PATCH  /api/item/categories/{id}
DELETE /api/item/categories/{id}
```

### ItemDefinition CRUD

```http
POST   /api/item/definitions
GET    /api/item/definitions
GET    /api/item/definitions/{key}
PATCH  /api/item/definitions/{id}
DELETE /api/item/definitions/{id}
```

Create example:

```json
{
  "name": "Gold",
  "alias": "gold",
  "categoryId": "currency-category-uuid",
  "description": "Universal currency",
  "stackable": true,
  "stackLimit": null,
  "holdLimit": null,
  "isActive": true
}
```

### Grant

```http
POST /api/item/grant
Authorization: Bearer ak_...
Content-Type: application/json

{
  "endUserId": "user-42",
  "grants": [
    { "definitionId": "gold-uuid", "quantity": 100 },
    { "definitionId": "gem-uuid", "quantity": 10 }
  ],
  "source": "admin_grant",
  "sourceId": "optional-idempotency-key"
}
```

**Response**:

```json
{
  "grants": [
    {
      "definitionId": "gold-uuid",
      "quantityBefore": 50,
      "quantityAfter": 150,
      "delta": 100
    },
    {
      "definitionId": "gem-uuid",
      "quantityBefore": 0,
      "quantityAfter": 10,
      "delta": 10
    }
  ]
}
```

### Deduct

```http
POST /api/item/deduct
Content-Type: application/json

{
  "endUserId": "user-42",
  "deductions": [
    { "definitionId": "gold-uuid", "quantity": 50 }
  ],
  "source": "exchange",
  "sourceId": "exchange-uuid"
}
```

Fails if the user's holdings are insufficient.

### Query inventory

```http
GET /api/item/users/{endUserId}/inventory
GET /api/item/users/{endUserId}/inventory?definitionId=uuid
```

### Query balance

```http
GET /api/item/users/{endUserId}/balance/{key}
```

`key` accepts UUID or alias.

**Response**:

```json
{
  "definitionId": "gold-uuid",
  "balance": 150
}
```

## Client API

### Query inventory

```http
GET /api/client/item/users/{endUserId}/inventory
x-api-key: cpk_...
x-user-hash: hmac_hex_string
```

`definitionId` query filter supported.

### Query balance

```http
GET /api/client/item/users/{endUserId}/balance/{key}
x-api-key: cpk_...
x-user-hash: hmac_hex_string
```

> Client API is read-only. Grants and deductions must come from admin APIs or from the check-in / exchange modules.

## holdLimit behavior

When a definition sets `holdLimit`:

- On grant, if the post-grant total exceeds `holdLimit`, the actual granted amount is truncated.
- e.g. `holdLimit=100`, user holds 90, grant 20 → actual grant = 10.
- `holdLimit=1` is great for unique items (achievement badges); re-granting doesn't increase the count.
