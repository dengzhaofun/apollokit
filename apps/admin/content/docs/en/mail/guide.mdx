> This page is for engineering teams. For product background and use cases, see the [Product Overview](./index).

The mail module handles "send a message to the player with optional reward attachments". It supports single-user, multi-user, and broadcast delivery, attachments (items / currency), revoke, expiry, and hard delete.

## Data model

- **message**: title, body, attachments, recipients, sent time, expiry.
- **user_state**: one row per `(message, endUserId)`, recording `read` / `claimed` times. For broadcast mail it's created lazily on the player's first inbox pull (avoids multi-million row blowup).

## Admin API

```http
POST   /api/mail/messages          send mail (broadcast / specified list)
GET    /api/mail/messages          list all mails for this org (with sent/read/claim counts)
GET    /api/mail/messages/{id}     single message detail
POST   /api/mail/messages/{id}/revoke   revoke (soft delete — invisible to player but preserved)
DELETE /api/mail/messages/{id}     hard delete (cascades user states)
```

## Client API

```http
GET  /api/client/mail/messages            player's inbox (with unread count)
GET  /api/client/mail/messages/{id}       single mail (does NOT auto-mark as read)
POST /api/client/mail/messages/{id}/read  explicitly mark read, idempotent
POST /api/client/mail/messages/{id}/claim claim attachment, idempotent
```

## Delivery strategies

- **Broadcast**: `targetType: 'all'` — visible to every player in this org. User_state rows are created lazily on each player's first inbox pull.
- **Multicast**: `targetType: 'list'` + `endUserIds: [...]`, inserts user_state rows in one batch.
- **Unicast**: multicast with one endUserId.

## Claim idempotency

`POST /messages/{id}/claim` internally runs `UPDATE ... WHERE claimed_at IS NULL ... RETURNING`. A second call returns `alreadyClaimed: true` without duplicating rewards. Attachments support multiple kinds:

- Items (via [item](../item) grant)
- Currency (via [currency](../currency) grant)
- Mixed combinations

Successful claim emits `mail.claimed`, consumable via [webhook](../webhooks).

## Expiry

After `expiresAt`, client `/claim` returns 409 `mail.expired`. The mail itself is not auto-deleted — claiming is just blocked. To purge, use `DELETE /messages/{id}`.

## Revoke vs hard delete

| Action | Player | Data |
|---|---|---|
| revoke | disappears | preserved (including read/claim stats) |
| DELETE | disappears | wiped (including cascade user_state) |

Revoke for ops mistakes, then hard-delete for cleanup after a while.
