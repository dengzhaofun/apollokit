> This page is for engineering teams. For product context and use cases, see the [Product Overview](./index).

The end-user module is "player profile metadata storage" and nothing more. It is **not** a player login system — ApolloKit is completely transparent about player identity. `endUserId` comes from your own game account system.

<Callout type="warn">
  `endUserId` is an opaque string. ApolloKit doesn't validate its format and doesn't care how your players log in. If you're tempted to use a Better Auth `userId` as an `endUserId` — stop. That's the SaaS admin id, not a player. See [authentication](../authentication).
</Callout>

## Data model

- **end_user_profiles**: `(orgId, endUserId)` unique, includes displayName / avatarUrl / metadata (JSONB).

## Admin API

```http
GET    /api/end-user/profiles
GET    /api/end-user/profiles/{endUserId}
PATCH  /api/end-user/profiles/{endUserId}
DELETE /api/end-user/profiles/{endUserId}   cascades all module data for this player
```

## Client API

```http
GET /api/end-user/profile
PUT /api/end-user/profile                   body: { displayName, avatarUrl, metadata }
```

## Why this module exists

- **Unified display info**: when guild lists, friend lists, and chat messages need a display name and avatar, there's one authoritative source.
- **Cascade anchor**: deleting a player triggers `ON DELETE CASCADE` across all module data (check-in, inventory, friends, leaderboard entries...) — no orphaned records. This is also the master switch for integration test environment cleanup.
- **Free-form metadata**: JSONB field stores anything — game account id, server, linked email, etc. ApolloKit stores it without interpreting it.

## Composing with other modules

All player-facing modules accept `endUserId` as the string you provide — they don't require the profile to exist. The profile only adds "a display name for the presentation layer."

## Events

`end-user.profile_updated` / `end-user.deleted` (the latter fires before the cascade, giving downstream systems a chance to invalidate caches).
