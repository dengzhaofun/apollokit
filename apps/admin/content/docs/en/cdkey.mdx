The CDKEY module supports "batch-issued codes, redeemable once or many times" flows. Ideal for channel packs, streamer codes, user compensation, and ad-campaign giveaways.

## Data model

- **batch**: metadata for a set of codes — validity window, per-code rewards (or per-code override), per-user/per-code limits, global redemption cap, reuse flags, etc.
- **code**: the actual string. Belongs to a batch, individually revocable.
- **log**: redemption audit (user, time, result, linked item/currency/mail id).

## Admin API

```http
# Batches
POST   /api/cdkey/batches                  create (no auto-generated codes)
GET    /api/cdkey/batches
GET    /api/cdkey/batches/{key}            by id or alias
PATCH  /api/cdkey/batches/{key}            update
DELETE /api/cdkey/batches/{key}            delete (cascades codes, states, logs)

# Codes
POST   /api/cdkey/batches/{batchId}/codes/generate   append-generate (repeatable, collision-free)
GET    /api/cdkey/batches/{batchId}/codes
PATCH  /api/cdkey/codes/{codeId}/revoke              revoke a single code

# Audit
GET    /api/cdkey/batches/{batchId}/logs             redemption logs for this batch

# Admin-side redeem (ops action)
POST   /api/cdkey/redeem                             body: { code, endUserId }
```

## Client API

```http
POST /api/client/cdkey/redeem   player redeem: { code }
```

Returns `{ rewards: [...], alreadyRedeemed: false }`.

## Code generation

`/codes/generate` generates N codes per call:

- `prefix`: e.g. `LAUNCH-`
- `length`: total length
- `charset`: defaults to uppercase letters + digits, excluding easily confused chars (O/0, I/1)
- `count`: how many to generate

Append mode means multiple generate calls accumulate on the same batch; existing codes are preserved and new ones are guaranteed unique within the batch.

## Redemption rules (per batch)

- `perCodeLimit`: redemptions per code (pack codes = 1; universal codes = N).
- `perUserLimit`: max codes one user can redeem in this batch (anti-farm).
- `globalLimit`: total redemptions per batch.
- `startAt` / `expireAt`: time window.
- `rewards`: batch-wide reward (or per-code `rewardOverride` at generation time).

Violations return 409 with codes like `cdkey.expired / cdkey.revoked / cdkey.user_limit_reached / cdkey.global_limit_reached / cdkey.already_redeemed`.

## Reward delivery path

On successful redemption, rewards are delivered to the player via [mail](./mail) (with attachments), not dropped straight into inventory — the player sees the inbox prompt, and it's auditable.

## Common scenarios

- **Streamer code**: `perUserLimit: 1`, `perCodeLimit: unlimited`, `globalLimit: 10000`, shared reward.
- **Physical CDKEY card**: one card = one code, `perCodeLimit: 1`. Generate in bulk, print on cards.
- **Ad campaign**: per-user unique code, `perCodeLimit: 1`, `rewardOverride` customized per user.

## Anti-abuse

`/redeem` goes through the standard client-credentials + HMAC, plus Upstash rate limiting (if enabled). The server temporarily blocks a user who hits too many `cdkey.invalid` redemptions within a day.
