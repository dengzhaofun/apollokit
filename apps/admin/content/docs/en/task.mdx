The task module turns any quantifiable player action (kills, cumulative logins, spend, shares…) into a configurable reward pipeline. Mental model: **business code just reports events; the task module decides who completed what**.

## Data model

- **category**: daily / weekly / achievement / activity task grouping. For display.
- **definition**: the full rules of a task — event name, field filters, target value, reset cadence, rewards, tier ladder.
- **assignment**: `(definition, endUserId)` pair — "this player is doing this task". Manual or bulk-default.
- **progress**: per-player × per-task value, auto-accumulated from events.

## Client API

```http
POST /api/client/task/events          report a business event (drives progress)
POST /api/client/task/list            list tasks + progress
POST /api/client/task/claim/{taskId}  claim a single-stage reward
POST /api/client/task/claim-tier      claim a specific tier in the ladder
```

## Event-driven progress

The key mechanism:

1. Business code calls `/api/client/task/events` when the player does something:
   ```json
   { "name": "kill_boss", "value": 1, "meta": { "bossId": "dragon" } }
   ```
2. The task module finds every definition with `eventName == 'kill_boss'`, applies each definition's `filter` (e.g. `meta.bossId == 'dragon'`), and increments the matching player's progress.
3. When progress hits `target`, `completedAt` is written; the player can claim the reward.

Filters use structured fields (`filter-fields.ts`), not string matching — this keeps definitions readable and lets ops build complex conditions in the dashboard.

## Tiered rewards (claim-tier)

A task can have multiple thresholds:

```
30 times → tier 1 (100 gold)
100 times → tier 2 (500 gold + bronze chest)
500 times → tier 3 (5000 gold + gold chest)
```

Each tier is claimable independently. The front-end shows "claim" when progress crosses a threshold.

## Activity integration

`event-forwarder.ts` forwards task-completion events into [activity](./activity)'s points system:

- Task complete → activity points += N → milestone unlocked

This lets tasks feed activity points without double-wiring in business code.

## Event output (for webhooks)

- `task.progress`: every progress change (high volume — can unsubscribe)
- `task.completed`
- `task.claimed`

`apps/server/src/modules/task/index.ts` declares the event schemas (`organizationId`, `endUserId`, `taskId`, `taskAlias`, `progressValue`, `completedAt`).

## Reset cadence

- `daily`: reset at 00:00 UTC (or the configured local timezone)
- `weekly`: reset on Monday
- `none`: never reset (achievements)

Reset clears `progressValue` only, not claim state — if you want daily rewards to reclaim each day, also tick "reset claim state" on the definition.
