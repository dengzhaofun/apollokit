The invite module implements "refer a friend, earn rewards": each player gets a unique invite code; when a new player registers with the code, both sides are linked in a referral relationship. When the new player meets a qualification condition (first purchase, reaching level N), the referrer can claim their reward.

## What problem does it solve?

Referrals are one of the lowest-cost acquisition channels, but building your own system requires handling a lot of details:

- Invite code generation and uniqueness guarantees
- A new player can only be referred by one existing player (prevent duplicate binding)
- When and how qualification conditions trigger
- Idempotent reward claiming (prevent double-claiming)
- Self-referral prevention

The invite module encapsulates all of this. The only business-side requirement: call one endpoint to mark "qualification reached" when the new player meets the criteria.

## Typical use cases

**Scenario 1: First purchase unlocks referrer reward**
After the new player makes their first purchase, the referrer can claim a reward. Business code calls "mark qualified" on payment success; the referrer sees a claimable reward in their invite page.

**Scenario 2: Level threshold unlocks referrer reward**
The referral relationship activates when the new player reaches level 10. Good for designs where rewards only trigger after the new player is genuinely engaged.

**Scenario 3: "Invite N friends" as a task**
"Invite 5 friends to register" configured as a task. The task system and invite module interoperate — when a new player binds, the referrer's task progress advances automatically.

**Scenario 4: Event-exclusive invite codes**
Generate limited invite codes for an event period; mark them invalid when the event ends, without affecting permanent codes.

## Core capabilities

- **Unique invite codes**: each player has a persistent code; voluntary rotation immediately invalidates the old one
- **Single binding**: a new player can only bind one referrer; redeeming the same code twice doesn't re-bind
- **Qualification gating**: new player meets the condition → business marks "qualified" → referrer can claim
- **Idempotent claiming**: same referral relationship can only be claimed once, concurrency-safe
- **Self-referral prevention**: players cannot redeem their own invite code
