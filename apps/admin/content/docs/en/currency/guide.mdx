> This page is for engineering teams. For product background and use cases, see the [Product Overview](./index).

The currency module is the backbone of the economy — any "deduct N gold / grant N gems" operation eventually records here. Within a single org you can define arbitrarily many currencies (gold, gems, stamina, season tokens…), each accounted for independently.

## Data model

- **Currency definition** (`/definitions`): metadata for each currency (id, alias, display name, icon, precision).
- **Wallet** (`/wallets`): balance record sharded by `(endUserId, currencyId)`. Lazily created on first grant.
- **Ledger** (`/ledger`): every grant/deduct appends one row (who, amount, reason, relatedId). Append-only.

## Admin API

```http
POST   /api/currency/definitions          create currency
GET    /api/currency/definitions          list
GET    /api/currency/definitions/{key}    by id or alias
PATCH  /api/currency/definitions/{id}     update
DELETE /api/currency/definitions/{id}     delete

GET    /api/currency/wallets?endUserId=   list all wallets for a user
GET    /api/currency/wallets/{endUserId}/{currencyId}
POST   /api/currency/wallets/grant        grant
POST   /api/currency/wallets/deduct       deduct
GET    /api/currency/ledger               query ledger (filter by time / user / currency / reason)
```

## Client API

```http
GET /api/client/currency/wallets
GET /api/client/currency/balance/{key}
```

Client can only **read** its own balance — never grant or deduct. All changes originate from the server (business side or other ApolloKit modules like shop / cdkey / exchange), which prevents front-end tampering.

## Grant atomicity

Under the hood, upserts use `INSERT ... ON CONFLICT ... DO UPDATE ... WHERE`, so concurrent grants never overwrite each other. Read/write separation: writes do not lock; reads go straight to primary.

## Combining with other modules

- [shop](../shop): purchases deduct from the wallet; if the product outputs another currency or item, it grants accordingly.
- [exchange](../exchange): prices quoted in currency; settlement via wallet.
- [cdkey](../cdkey): redemption can grant currency directly.
- [mail](../mail): a mail attachment like "1000 gold" is simply a grant.
- [storage-box](../storage-box): deposits/withdrawals flow through the wallet.

## Gotchas

- Precision is set via `definition.precision` (default integer). Don't mix decimal and integer currencies carelessly.
- No "negative grant" semantics — use `/deduct` to deduct. Don't pass negative values to `grant` or the ledger reason will be misleading.
- Ledger is append-only, no "edit history" API. For corrections, issue opposing `deduct` + `grant` entries with `reason: correction`.
