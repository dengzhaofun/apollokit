The friend module covers the full "add friend / receive requests / reject / block" social graph maintenance. Relationships are bidirectional; blocks are unidirectional; all limits are per-tenant configurable.

## Data model

- **friend_settings** (one row per org): `maxFriends` / `maxBlocked` / `maxRequests`, request expiry policy.
- **friend_relationships**: confirmed bidirectional friendships.
- **friend_requests**: pending / accepted / rejected / expired state transitions.
- **friend_blocks**: unidirectional block list.

## Admin API

```http
PUT /api/friend/settings                    per-org limits/policy
GET /api/friend/settings
```

## Client API

```http
POST /api/friend/end-user/request           send request
POST /api/friend/end-user/request/{id}/accept
POST /api/friend/end-user/request/{id}/reject
POST /api/friend/end-user/request/{id}/cancel
DELETE /api/friend/end-user/{friendId}      remove friend

POST /api/friend/end-user/block
DELETE /api/friend/end-user/block/{userId}

GET /api/friend/end-user/list
GET /api/friend/end-user/requests
GET /api/friend/end-user/blocks
```

## Business rules

- **Bidirectional**: on accept, two mirror rows are written to `friend_relationships` so both sides match in queries; deletion removes both.
- **Block**: one-way. A blocks B → A doesn't see B's requests; B may still see A (configurable asymmetry).
- **Limits**: friends / blocks / pending requests are capped by settings. Hitting the cap returns `LIMIT_EXCEEDED`.
- **Request expiry**: per `requestTtl`, pending requests auto-transition to `expired`.

## Combining with other modules

- [friend-gift](./friend-gift): send daily gifts between friends.
- [invite](./invite): auto-friend after successful invite (optional).
- [leaderboard](./leaderboard): `scope=friend` for friend-only boards.

## Events

`friend.requested` / `friend.accepted` / `friend.removed` / `friend.blocked`.
