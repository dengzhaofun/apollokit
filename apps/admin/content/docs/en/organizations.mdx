ApolloKit's multi-tenancy model is built on **Better Auth's organization plugin**. The entire data plane is sliced by `organizationId`: items, configs, orders, event streams, credentials — everything belongs to exactly one org, and cross-org reads/writes are rejected server-side.

## Mental model

An ApolloKit admin account (admin user) can:

- Create multiple organizations (one company / one game / one environment == one org)
- Invite other admins per org and assign roles
- Issue per org:
  - Admin **API Keys** (scripts calling the admin API)
  - **Client Credentials** (game client use)

"Which org am I operating on" comes from `activeOrganizationId` in the session — switching orgs switches context, and every business API call implicitly carries that id.

## What belongs in an org

| Granularity | Works | Doesn't |
|---|---|---|
| One org per game project | ✅ Recommended. Different games' configs/economies stay fully isolated. | — |
| Split staging / prod into 2 orgs | ✅ Recommended. Data never bleeds. | — |
| One org per region / shard | ❌ Use business fields on [end-user](./end-user) for that. | |
| One org per player | ❌ That's end-user's job, not org's. |

## Members & roles

Better Auth organization ships standard roles:

| Role | Capability |
|---|---|
| `owner` | Everything, incl. disband & transfer ownership |
| `admin` | Member management, all module CRUD, issue credentials |
| `member` | Read configs, run ops (send mail, publish shop items), but can't change infra |

Invite flow: Dashboard → "Team" → enter email + role → system sends invite → the invitee clicks the link to join.

## Switching active organization

Use the top bar switcher → Better Auth writes `activeOrganizationId` on the session row. After switching, every admin API call targets the new org.

API side:

```http
POST /api/auth/organization/set-active
Content-Type: application/json

{ "organizationId": "org_..." }
```

## Credentials & scope

Both kinds of credentials are **strictly scoped to one org**:

| Credential | Used by | Scope |
|---|---|---|
| Admin API Key (`ak_*`) | Scripts / CI / your back-office calling the admin API | Bound at issuance, immutable |
| Client Credentials (`cpk_*` + `csk_*`) | Game client calling `/api/client/*` | Bound at issuance |

One `cpk` can only operate on data inside its bound org. Want to integrate a second org? Issue a new pair.

### Pitfalls to avoid

- **Don't put a prod cpk in a test env.** Use two orgs (or two cpks with `devMode` in the same org).
- **Don't share a cpk across projects.** Event streams, leaderboards, and mail end up entangled.

## Cross-tenant safety

The server's `requireClientUser` middleware includes a guard: if the end-user logged in via Channel A (Better Auth session) and that session's `organizationId` doesn't match the `cpk`'s org, it returns `session_tenant_mismatch` (403). This prevents an attacker from pairing a session cookie from org A with a cpk from org B.

## Delete & archive

- Disable a member → Better Auth's `disable` (audit preserved, cannot log in)
- Remove a member → delete the `member` row
- Delete an org → no one-click delete today (cascade volume is large). Contact the maintainer for data export + deletion.

## FAQ

**How does one admin account join multiple orgs?**
Better Auth's `user` is cross-org. One user can appear in multiple `member` rows, each with an org-specific role.

**How to hand off an org to someone new?**
The current owner invites the newcomer as admin → the owner transfers ownership → self-demotes to admin or leaves.

**Are limits per user or per org?**
Per org (rate limits, storage, event volume). Upstash rate-limit keys include `organizationId`.
