The exchange module manages trade relationships between items, supporting per-user and global execution limits.

## Core concepts

### ExchangeConfig

A container for a set of exchange options (e.g. "Spring Festival Exchange"):

| Field | Type | Meaning |
|------|------|---------|
| `name` | string | Config name |
| `alias` | string? | Org-unique alias |
| `isActive` | boolean | Enable flag |
| `metadata` | object? | Custom extension data |

### ExchangeOption

Each option defines the "cost" and "reward" of a single exchange:

| Field | Type | Meaning |
|------|------|---------|
| `name` | string | Option name |
| `costItems` | array | Cost items `[{definitionId, quantity}]` |
| `rewardItems` | array | Reward items `[{definitionId, quantity}]` |
| `userLimit` | number? | Per-user exec cap (`null` = unlimited) |
| `globalLimit` | number? | Global exec cap (`null` = unlimited) |
| `globalCount` | number | Current global count (read-only) |
| `sortOrder` | number | Sort weight |
| `isActive` | boolean | Enable flag |

### ExchangeUserState

Tracks per-user counts per option:

| Field | Meaning |
|------|---------|
| `optionId` | Option id |
| `endUserId` | End-user id |
| `count` | Times executed |

## Admin API

### ExchangeConfig CRUD

```http
POST   /api/exchange/configs
GET    /api/exchange/configs
GET    /api/exchange/configs/{key}
PATCH  /api/exchange/configs/{id}
DELETE /api/exchange/configs/{id}
```

### ExchangeOption CRUD

```http
POST   /api/exchange/configs/{configKey}/options
GET    /api/exchange/configs/{configKey}/options
PATCH  /api/exchange/options/{optionId}
DELETE /api/exchange/options/{optionId}
```

Create example:

```json
{
  "name": "100 gold → 10 gems",
  "costItems": [
    { "definitionId": "gold-uuid", "quantity": 100 }
  ],
  "rewardItems": [
    { "definitionId": "gem-uuid", "quantity": 10 }
  ],
  "userLimit": 5,
  "globalLimit": 1000,
  "sortOrder": 1,
  "isActive": true
}
```

### Admin-side execution

```http
POST /api/exchange/options/{optionId}/execute
Content-Type: application/json

{
  "endUserId": "user-42",
  "idempotencyKey": "unique-request-id"
}
```

### Query user state

```http
GET /api/exchange/options/{optionId}/users/{endUserId}/state
```

## Client API

### Execute exchange

```http
POST /api/client/exchange/execute
x-api-key: cpk_...
Content-Type: application/json

{
  "optionId": "option-uuid",
  "endUserId": "user-42",
  "userHash": "hmac_hex_string",
  "idempotencyKey": "unique-request-id"
}
```

**Response**:

```json
{
  "success": true,
  "exchangeId": "uuid",
  "optionId": "option-uuid",
  "costItems": [
    { "definitionId": "gold-uuid", "quantity": 100 }
  ],
  "rewardItems": [
    { "definitionId": "gem-uuid", "quantity": 10 }
  ]
}
```

## Execution flow

```
1. Client calls the execute API
2. Server validates:
   ├─ user count < userLimit?
   ├─ global count < globalLimit?
   └─ user inventory has enough costItems?
3. Atomic execution on success:
   ├─ deduct costItems (via item module's deductItems)
   ├─ grant rewardItems (via item module's grantItems)
   ├─ increment user count
   └─ increment global count
4. Return exchange result
```

### Limit errors

When a cap is hit, returns `LIMIT_EXCEEDED`:

```json
{
  "error": "User exchange limit exceeded",
  "code": "LIMIT_EXCEEDED"
}
```

### Idempotency

Pass `idempotencyKey` to prevent double-execution. Repeated calls with the same key return the previous result instead of executing again.
