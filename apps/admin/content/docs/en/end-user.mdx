The end-user module is a "player profile metadata store" — nothing more. It is **not** a player login system. ApolloKit is transparent to player identity: `endUserId` comes from your own game account system.

<Callout type="warn">
  `endUserId` is an opaque string; ApolloKit neither validates its format nor manages how players log in. If you are about to stuff Better Auth's `userId` into `endUserId`, stop — that's the SaaS admin id, not the player id. See [authentication](./authentication).
</Callout>

## Data model

- **end_user_profiles**: unique on `(orgId, endUserId)`, with displayName / avatarUrl / metadata (JSONB).

## Admin API

```http
GET    /api/end-user/profiles
GET    /api/end-user/profiles/{endUserId}
PATCH  /api/end-user/profiles/{endUserId}
DELETE /api/end-user/profiles/{endUserId}   cascade-clears this player's data across all modules
```

## Client API

```http
GET /api/end-user/profile
PUT /api/end-user/profile                   body: { displayName, avatarUrl, metadata }
```

## Why this module exists

- **Canonical display info**: guild lists, friend lists, chat messages need a unified source of player nickname/avatar.
- **Cascade point**: deleting a player triggers `ON DELETE CASCADE` across every module (check-in, inventory, friends, leaderboards…), avoiding orphans. This is also how integration tests reset state.
- **Free-form metadata**: JSONB field for whatever — game account id, shard, linked email. ApolloKit doesn't interpret, just stores.

## Combining with other modules

All player-facing modules assume `endUserId` is the string you provide and do not require a profile to exist. Profile is only "give the display layer a name".

## Events

`end-user.profile_updated` / `end-user.deleted` (the latter fires pre-cascade so downstream can flush caches).
