ApolloKit's multi-tenancy model is built on **Better Auth's organization plugin** (which we surface to users as "projects"). The entire data plane is sliced by `organizationId`: items, configs, orders, event streams, credentials â€” everything belongs to exactly one project, and cross-project reads/writes are rejected server-side.

## Mental model

An ApolloKit admin account (admin user) can:

- Create multiple projects (one company / one game / one environment == one project)
- Invite other admins per project and assign roles
- Issue per project:
  - Admin **API Keys** (scripts calling the admin API)
  - **Client Credentials** (game client use)

"Which project am I operating on" comes from `activeOrganizationId` in the session â€” switching projects switches context, and every business API call implicitly carries that id.

## Members & roles

Better Auth organization ships standard roles:

| Role | Capability |
|---|---|
| `owner` | Everything, incl. disband & transfer ownership |
| `admin` | Member management, all module CRUD, issue credentials |
| `member` | Read configs, run ops (send mail, publish shop items), but can't change infra |

Invite flow: Dashboard â†’ "Team" â†’ enter email + role â†’ system sends invite â†’ the invitee clicks the link to join.

## Switching the active project

Use the top bar switcher â†’ Better Auth writes `activeOrganizationId` on the session row. After switching, every admin API call targets the new project.

API side:

```http
POST /api/auth/organization/set-active
Content-Type: application/json

{ "organizationId": "org_..." }
```

## Credentials & scope

Both kinds of credentials are **strictly scoped to one project**:

| Credential | Used by | Scope |
|---|---|---|
| Admin API Key (`ak_*`) | Scripts / CI / your back-office calling the admin API | Bound at issuance, immutable |
| Client Credentials (`cpk_*` + `csk_*`) | Game client calling `/api/client/*` | Bound at issuance |

One `cpk` can only operate on data inside its bound project. Want to integrate a second project? Issue a new pair.

### Pitfalls to avoid

- **Don't put a prod cpk in a test env.** Use two projects (or two cpks with `devMode` in the same project).
- **Don't share a cpk across projects.** Event streams, leaderboards, and mail end up entangled.

## Cross-tenant safety

The server's `requireClientUser` middleware includes a guard: if the end-user logged in via Channel A (Better Auth session) and that session's `organizationId` doesn't match the `cpk`'s project, it returns `session_tenant_mismatch` (403). This prevents an attacker from pairing a session cookie from project A with a cpk from project B.

## Delete & archive

## FAQ

**How does one admin account join multiple projects?**
Better Auth's `user` is cross-project. One user can appear in multiple `member` rows, each with a project-specific role.

**How to hand off a project to someone new?**
The current owner invites the newcomer as admin â†’ the owner transfers ownership â†’ self-demotes to admin or leaves.

**Are limits per user or per project?**
Per project (rate limits, storage, event volume). Upstash rate-limit keys include `organizationId`.
