组队模块负责「玩一局临时组一支队」的短生命周期小队伍：创建队伍、邀请成员、开始游戏、解散，和公会的「长期组织」是两套不同的概念。

## 解决什么问题

多人游戏中，玩家在进入匹配或副本之前往往需要先组成小队。自己实现组队系统需要处理：

- 并发邀请导致人数超限（两人同时被邀请，一人加入后另一人再加就超了）
- 队长离开后谁来接管，如何自动解散
- 队伍状态管理（招募中 / 已满 / 游戏中），防止重复开始
- 快速匹配时如何把散人安排到同一个队伍

组队模块把这些逻辑封装好，业务只需关注「进游戏」这个核心动作。

## 典型使用场景

**场景 1：多人副本组队**
玩家创建队伍（4 人上限），发邀请链接给好友，好友接受后进入队伍，队长确认 4 人齐后按「开始游戏」，队伍状态变为 in_game，进入副本。

**场景 2：快速匹配**
不想组队的玩家直接点「快速匹配」，系统自动将其分配到一支人数不满的队伍，或为其创建新队伍等待其他人加入。

**场景 3：队长离开后继任**
队长意外离线，系统按配置（`autoDissolveOnLeaderLeave=false`）自动将最先加入的成员晋升为新队长，队伍继续。

**场景 4：天梯排位组队**
天梯匹配前先用组队模块组好固定队伍，再将整个队伍送进天梯匹配池，保证队友间平分匹配权重。

## 核心能力

- **临时队伍管理**：创建、邀请、接受/拒绝、踢人、解散，全流程 API
- **状态管理**：open（招募中）/ closed（已满）/ in_game（游戏中）/ dissolved（已解散）
- **并发安全**：版本字段防止并发超编，冲突时返回错误由调用方重试
- **队长策略**：队长离开时可配置自动解散或自动继任
- **快速匹配**：可选开启，将散人自动分配到合适队伍

## 适合谁

| 角色 | 关注点 |
|------|--------|
| 决策者 / 老板 | 多人体验是游戏核心竞争力，组队模块降低实现门槛 |
| 产品经理 | 灵活配置队伍大小、邀请过期时间、匹配策略 |
| 运营人员 | 可查看活跃队伍数量，辅助分析多人内容的活跃度 |
| 研发团队 | 标准 API，并发安全由模块保证，不用自己实现版本锁 |

## 与其他模块的关系

- **[公会](../guild)**：公会成员之间优先组队（可先过滤公会成员再做快速匹配）
- **[天梯](../rank)**：天梯匹配前先组好固定队伍，再整队进入匹配池
- **[排行榜](../leaderboard)**：队伍维度的积分榜（如队伍击杀数、副本通关速度）
