物品系统是整个平台的**资源基础层**——你在游戏里能发给玩家的一切东西，无论是金币、道具、皮肤还是徽章，都由它统一管理。签到奖励、兑换、抽奖发奖，最终都调用物品系统完成资源到账。

## 解决什么问题

游戏后台里最容易"各自为政"的就是物品数据——签到奖励写死在签到表里，抽奖奖励存在抽奖表里，客服扣除走另一条通道。结果：

- 想查玩家某个物品的余额，要跑五张表
- 想加一种新道具，要改多个模块的代码
- 发错了奖品，没有统一的审计日志可查

物品系统解决这三个问题：**单一数据源 + 统一发放通道 + 完整审计链**。

## 典型使用场景

**场景 1：运营发奖**
节日活动结束后，运营人员在管理台选定用户群，一键批量发放「节日礼盒」。物品系统记录每一笔，发放记录可按来源/时间检索。

**场景 2：多模块联动**
签到第 7 天奖励 100 金币 + 1 个道具箱。签到模块调物品系统的发放接口，物品系统负责落库和日志，签到模块不需要关心背包实现。

**场景 3：唯一物品（成就徽章）**
将 `holdLimit=1` 的物品重复发给同一玩家，系统自动截断——徽章不会叠加，也不会报错。

**场景 4：堆叠限制（宝箱）**
每个格子最多放 99 个，满了自动新开一格。前端直接用 InventoryView 渲染背包，无需自己实现分堆逻辑。

## 核心能力

- **物品分类**：按业务分组（货币、道具、材料），方便前端筛选展示
- **堆叠规则**：支持无限堆叠、有限堆叠、不可堆叠三种模式
- **持有上限**：`holdLimit` 限制玩家最多持有量，适合唯一物品
- **批量发放 / 扣除**：单次请求发多种物品，支持幂等键防重复
- **余额查询**：按物品 alias（如 `gold`）或 UUID 查单个物品的持有数量
- **审计日志**：每次变动自动记录来源、变动量、变动前后余额，客服可追溯

## 适合谁

| 角色 | 关注点 |
|------|--------|
| 决策者 / 老板 | 所有资源变动有记录可查，风控和审计有基础 |
| 产品经理 | 新增物品类型无需改代码，管理台配置即生效 |
| 运营人员 | 管理台直接发奖、查余额、导出发放日志 |
| 研发团队 | 一套接口搞定所有物品操作，不用为每种奖励单独建表 |

## 与其他模块的关系

- **[签到](../check-in)** / **[抽奖](../lottery)** / **[兑换](../exchange)**：发奖时调物品系统
- **[货币](../currency)**：货币是物品的特殊形态，大额流水优先走货币模块
- **[邮件](../mail)**：邮件附件通过物品系统发放，玩家领取时到账
