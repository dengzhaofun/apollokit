实体模块是一个「面向角色 / 怪物 / 装备通用数据的建模层」：把游戏里「有属性、可升级、可换皮、可编队」的单位全部抽象成 entity，用 schema 定义属性结构，用 blueprint 定义具体模板，用 skin 叠加属性加成，用 formation 约束阵容规则。

## 解决什么问题

游戏内容管理的常见问题：

- 角色、怪物、装备各自定义一套属性结构，相互不通用，维护成本高
- 皮肤只是换贴图，还是要带属性加成？要不要支持多皮肤叠加？
- 阵容的槽位类型限制、重复上阵规则需要在客户端和服务端各写一遍

实体模块提供通用建模层：一套 schema 可以服务多种单位类型，属性分层计算（蓝图 → 等级加成 → 皮肤加成），阵容校验在服务端统一执行。

## 典型使用场景

**场景 1：英雄角色系统**
定义 `hero` schema（HP、ATK、DEF、SPEED 等字段），每个英雄是一个 blueprint，玩家获得英雄即创建 instance，升级时 instance 上的等级字段变化，属性通过分层计算实时得出。

**场景 2：装备与皮肤**
武器 blueprint 上挂多个皮肤，不同皮肤带不同属性加成。玩家切换皮肤后，装备的实际属性实时更新。

**场景 3：阵容编排**
定义 5 人阵容，前排 2 槽只接受类型为「战士」或「坦克」的英雄，后排 3 槽接受任意类型。保存阵容时服务端校验，不合规的阵容配置直接拒绝。

**场景 4：怪物数据管理**
关卡里的怪物统一用 `monster` schema 管理，设计师在管理台调整属性，客户端拉取后直接渲染，无需发版。

## 核心能力

- **Schema 定义**：用字段声明属性结构（HP/ATK/标签），一个 schema 可以服务多种单位
- **Blueprint 模板**：具象单位模板，引用 schema 并填写具体属性值
- **皮肤系统**：皮肤挂在 blueprint 上，支持叠加属性加成
- **实例管理**：玩家拥有的具体单位（含等级、状态）
- **阵容校验**：槽位类型限制、重复上阵规则在服务端统一校验

## 适合谁

| 角色 | 关注点 |
|------|--------|
| 决策者 / 老板 | 内容管理统一化，设计师可直接在管理台调整数值 |
| 产品经理 | 通用建模层降低新单位类型的开发成本 |
| 策划 / 运营 | 管理台直接调整 blueprint 属性，无需改代码 |
| 研发团队 | 不用为每种单位类型写单独的 CRUD，接一套 API |

## 与其他模块的关系

- **[关卡](../level)**：通关可解锁新的实体蓝图
- **[物品系统](../item)**：获得实体 = 发一个道具，道具后处理器调实体模块创建实例
- **[图鉴](../collection)**：获得新 blueprint 实例 → 解锁图鉴条目
