埋点分析模块是 ApolloKit 和 [Tinybird](https://tinybird.co/) 的对接层。ApolloKit 自己不做 OLAP，事件数据照常通过 [webhooks](../webhooks) 落到下游（或直接推 Tinybird Events API），这个模块的作用是**为租户签发可以直接查 Tinybird 的 scoped token**，让管理台看板不经过自己的后端就能拉数据。

## 数据模型

这个模块没有自己的表——所有配置在 `wrangler.jsonc` 的环境变量里（`TINYBIRD_ADMIN_TOKEN`、`TINYBIRD_PIPE_PREFIX` 等），多租户隔离由 token 里的 `org_id` scope 保证。

## 管理端 API

```http
POST /api/analytics/token                  body: { ttlSeconds? }
                                            → { token, expiresAt }
```

- 默认 TTL **600 秒**（10 分钟）。
- 返回的 token 只能查本 org 的数据——Tinybird 端的 pipe 用 `{{ String(org_id, required=True) }}` 参数锁死，拿到 token 的前端没法越权。

没有客户端 API——玩家端不直接查分析数据。

## 业务规则

- **不可否认**：token 里 org_id 在服务端签发时写死，客户端没法改（HMAC-SHA256 签名）。
- **短 TTL**：不要把 TTL 拉长。如果真需要长看板会话，前端定期 refresh token 即可。
- **防 fanout 事件丢失**：ApolloKit 服务端事件先写 Postgres 的 `event_history` 表，再异步推 Tinybird。Tinybird 临时不可达时不丢数据。

## 前端用法

```ts
// 前端先向 ApolloKit 要一个 token
const { token } = await fetch('/api/analytics/token', { method: 'POST' }).then(r => r.json())

// 然后用这个 token 直连 Tinybird
const res = await fetch('https://api.tinybird.co/v0/pipes/daily_active_users.json', {
  headers: { Authorization: `Bearer ${token}` },
})
```
