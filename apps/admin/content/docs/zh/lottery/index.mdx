抽奖模块提供「花消耗品换随机奖励」的完整实现。无论是转盘、宝箱还是抽卡，都支持通过配置奖池、档位、概率和保底规则来实现——不需要每次重写随机逻辑。

## 解决什么问题

抽奖是游戏里最复杂的发奖机制之一：

- 多档奖品（SSR/SR/R）各自有不同概率，档内各奖品还有权重
- 保底机制：连续 N 抽未出高档，第 N+1 抽必出（抽卡必备）
- 每次抽奖要扣费、记录、发奖，三步要原子
- 抽奖记录要可查，避免「没有保底」的纠纷

这些逻辑写起来并不难，但要做对需要很多细节：服务端随机（防客户端作弊）、保底计数并发安全、发奖幂等。抽奖模块提供**经过验证的实现**，直接配置用。

## 典型使用场景

**场景 1：抽卡（Gacha）**
角色/武器池：SSR 档 1%，SR 档 10%，R 档 89%。配置保底：连续 89 抽无 SSR，第 90 抽必出 SSR。多抽可设折扣（10 连 = 9 连价格）。

**场景 2：宝箱**
开箱玩法：扁平奖品池，各物品直接设概率。无档位，无保底。玩家消耗活动道具开箱，随机得到奖品。

**场景 3：活动专属转盘**
节日活动限时转盘，只在活动期间开放，作为活动（activity）的子节点挂载。活动结束，转盘关闭。玩家消耗活动积分而非通用货币。

**场景 4：全勤奖随机池**
签到 7 天获得一次抽取机会，随机从「稀有物品池」里抽一件。消耗品是虚拟的「抽奖机会」，而不是货币。

## 核心能力

- **档位 + 奖品**：支持多档（SSR/SR/R），档内按权重随机；也支持无档位的扁平模式
- **保底规则**：配置阈值和必出档位，累计计数自动追踪，达到阈值后触发
- **服务端随机**：使用 `crypto.getRandomValues`，结果无法被客户端预测
- **单抽 / 多抽**：多抽支持折扣配置
- **完整审计**：每次抽奖记录档位、奖品、保底计数、消耗，可查历史
- **活动集成**：可作为活动子节点，使用活动积分作为消耗品

## 适合谁

| 角色 | 关注点 |
|------|--------|
| 决策者 / 老板 | 保底机制让玩家有确定性预期，降低弃坑风险；审计记录防止「运气争议」 |
| 产品经理 | 灵活配置各类随机玩法，无需为每种变体单独开发 |
| 运营人员 | 管理台配置奖池、查看各奖品的实际出率和保底触发情况 |
| 研发团队 | 接入抽奖接口，不用自己实现随机算法、保底计数和并发安全 |
