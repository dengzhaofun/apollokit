终端玩家模块是一个「玩家档案元信息存储」，仅此而已。它不是登录系统——ApolloKit 对玩家身份完全透明，`endUserId` 来自你自己的游戏账号体系。

## 解决什么问题

ApolloKit 的所有面向玩家的模块都需要一个 `endUserId`，但各个展示场景（公会成员列表、好友列表、排行榜）需要的不只是一个 id，还需要昵称和头像。如果每个模块各存一份，就会出现数据不一致。

端玩家模块提供统一的展示信息来源，同时也是级联删除的锚点：删除一个玩家，该玩家在所有模块的数据（签到、背包、好友关系、排行榜……）都通过 `CASCADE` 自动清除。

<Callout type="warn">
  `endUserId` 是不透明字符串，ApolloKit 不校验其格式，也不管玩家怎么登录。这个 id 来自你自己的游戏账号体系，不是 ApolloKit 的 SaaS 管理员 id。
</Callout>

## 典型使用场景

**场景 1：展示昵称和头像**
公会成员列表、排行榜需要展示玩家昵称和头像。profile 是统一来源，不需要每个模块各存一份展示信息。

**场景 2：玩家注销**
玩家申请注销账号，删除 end-user profile 即可通过 CASCADE 清除该玩家在所有模块的全部数据，符合 GDPR 等数据删除要求。

**场景 3：自定义扩展信息**
需要跟 `endUserId` 关联存储的游戏业务数据（服务器区、VIP 等级、绑定邮箱等）放在 `metadata` JSONB 字段，ApolloKit 只存不解读。

**场景 4：测试环境清理**
集成测试时创建测试玩家，测试完成后删除 end-user profile，所有关联数据一并清除，测试环境干净。

## 核心能力

- **统一展示信息**：昵称、头像 URL，所有模块的展示层共用
- **自由 metadata**：JSONB 字段存储任意业务扩展数据
- **级联删除**：删除 profile 触发所有模块数据的 CASCADE 清理
- **ID 透传**：不校验 `endUserId` 格式，兼容任何游戏账号体系

## 适合谁

| 角色 | 关注点 |
|------|--------|
| 决策者 / 老板 | 玩家数据统一入口，注销请求可一键处理，满足合规要求 |
| 产品经理 | 理解平台玩家模型的边界：ApolloKit 只管运营数据，不管登录 |
| 运营人员 | 管理台可查看和编辑玩家档案，处理客诉时有完整数据视图 |
| 研发团队 | 一次对接，其余模块的 `endUserId` 直接传自己系统的 uid |

## 与其他模块的关系

所有面向玩家的模块都依赖 `endUserId`，但不依赖 profile 存在。Profile 只是「给展示层一个昵称和头像」的可选数据。
