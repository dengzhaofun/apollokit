好友模块提供游戏内完整的社交关系维护：发送好友请求、接受/拒绝、删除好友、拉黑屏蔽。好友关系双向、拉黑单向，所有上限可按项目灵活配置。

## 解决什么问题

没有好友系统，玩家之间是陌生人。但自建好友系统要处理很多边界情况：

- 双向关系如何保持一致（A 删了 B，B 那边要同步删）
- 拉黑后能否互相看到、能否收到对方的好友请求
- 每人最多几个好友、未处理请求上限是多少
- 请求长时间未回复要不要自动过期

好友模块把这些规则全部封装好，接入一次，后续通过配置调整。

## 典型使用场景

**场景 1：经典好友系统**
玩家互相发请求、接受后成为好友，可以互相查看在线状态、组队、送礼。

**场景 2：拉黑/屏蔽**
玩家拉黑某人后，对方的好友请求不再出现，双方交互受限，且拉黑关系对被拉黑方不可见。

**场景 3：好友上限运营调控**
不同阶段的游戏可能需要不同的好友上限——早期测试可能设 50，正式运营后扩展到 200——通过配置实时调整，无需改代码。

**场景 4：社交网络衍生玩法**
好友关系作为其他模块的前提：好友间送礼、好友内部排行榜、仅好友可见的战绩——都依托好友模块的关系数据。

## 核心能力

- **双向关系管理**：好友请求、接受、拒绝、撤销、删除，关系严格双向一致
- **单向拉黑**：屏蔽某玩家，被屏蔽方不知情，可配置不对称可见规则
- **好友上限**：每项目可配置好友上限、拉黑上限、未处理请求上限
- **请求自动过期**：未回复的好友请求按配置的 TTL 自动转为过期状态

## 适合谁

| 角色 | 关注点 |
|------|--------|
| 决策者 / 老板 | 好友系统是留存与社交裂变的基础，接入一次，后续无需重复开发 |
| 产品经理 | 灵活配置好友上限、请求过期规则等参数，适应不同产品阶段 |
| 运营人员 | 可查看玩家的好友关系网络，协助处理社交纠纷 |
| 研发团队 | 调用标准 API，无需自己实现双向一致性与并发安全逻辑 |
