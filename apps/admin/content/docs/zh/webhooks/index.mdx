ApolloKit 把所有业务动作（签到成功、兑换完成、任务达成、抽奖结果……）抽象成规范化事件，通过事件中心统一管理。事件既写进内部审计日志，也可以经 Webhook 推送给你的下游系统，让 ApolloKit 成为整个运营体系的事件总线。

## 解决什么问题

游戏运营系统往往是孤岛：

- 玩家完成任务 → 想同步到第三方 CRM / 推送平台 → 需要自己在业务里写联动代码
- 活动里程碑领奖 → 想触发外部数据统计 → 又是一段单点逻辑
- 随着模块增加，「事件 → 外部系统」的联动代码越来越多，维护成本线性增长

事件中心把「产生事件」和「消费事件」解耦：ApolloKit 产生标准化事件，你的外部系统通过 Webhook 订阅，联动代码只在外部系统里写一次。

## 典型使用场景

**场景 1：玩家行为同步到 CRM**
`task.completed` 事件触发时，推送玩家 id 和任务信息到外部 CRM，CRM 据此更新玩家标签，触发下一步推送策略。

**场景 2：活动结算触发外部奖励**
`activity.milestone_claimed` 事件推送到自研结算系统，结算系统根据事件数据给玩家发放实物奖品或兑换码。

**场景 3：实时监控与告警**
订阅 `lottery.drawn` 事件，如果某个时间窗口内 SSR 出货率明显偏低，触发告警给运营团队查排查概率配置。

**场景 4：数据分析管道**
所有事件推送到数据仓库入库，配合埋点分析模块构建完整的 BI 数据流。

## 核心能力

- **统一事件目录**：所有模块的事件在一处查看，含字段 schema 和最近出现时间
- **Webhook 订阅**：为特定事件名（或前缀匹配）配置接收 endpoint
- **签名校验**：每次投递带 HMAC 签名，接收方可验证真实性，防伪造
- **幂等 id**：每个事件有唯一 id，重试时同一事件可能投递多次，接收方应去重
- **指数退避重试**：失败时自动重试，最多 8 次，跨 24 小时

## 适合谁

| 角色 | 关注点 |
|------|--------|
| 决策者 / 老板 | ApolloKit 可以作为整个运营系统的事件总线，减少孤岛 |
| 产品经理 | 理解平台能产出哪些事件，规划下游联动 |
| 运营人员 | 通过事件驱动外部工具，无需研发每次都写联动代码 |
| 研发团队 | 订阅一次，所有事件都走同一套 Webhook 接收逻辑 |

## 与其他模块的关系

每个功能模块都会产生事件，具体事件名见各模块文档。以下是常见的事件消费场景：

- **[埋点分析](../analytics)**：事件进分析引擎，构建实时看板
- **[任务系统](../task)**：`task.completed` 是最常被订阅的事件之一
- **[活动](../activity)**：活动里程碑领奖事件驱动外部结算
