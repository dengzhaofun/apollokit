任务系统把「玩家做了某件事」自动转化为「任务进度推进 + 奖励发放」。业务侧只负责上报事件，任务系统决定哪些任务因此完成、发放什么奖励。覆盖日常任务、周常任务、成就、活动任务等所有场景。

## 解决什么问题

每个游戏里的任务系统往往长这样：

- 前端上报「击杀了一个 Boss」
- 后端查所有相关任务，看哪些的条件满足了
- 满足了就更新进度，达成就发奖
- 有些任务每天重置，有些永不重置
- 有些任务有阶梯：30 次领第一档，100 次领第二档

这套逻辑每个游戏都要做，每次做都差不多，但每次都要重写。任务系统把它变成**配置**：定义任务名称、监听什么事件、目标值是多少、发什么奖励，剩下的自动完成。

## 典型使用场景

**场景 1：日常任务**
「每日击杀 10 个怪物」「每日登录」「每日消费 X 货币」。每天重置，完成即发奖，简单直接。玩家每天有一批小任务要做，驱动日活。

**场景 2：成就**
「累计击杀 1000 个怪物」「首次组队」「完成 50 个副本」。永不重置，多级阶梯（500 次、1000 次、5000 次各发一档奖励）。成就激励长线玩家持续游玩。

**场景 3：活动任务**
「活动期间完成 3 个地下城」「邀请 1 名好友加入」。活动结束后任务停用，历史进度保留。任务完成触发活动积分，联动里程碑发奖。

**场景 4：引导任务**
「完成新手教程」「首次进入公会」「进行首次充值」。一次性，完成即永久标记，不再显示。

## 核心能力

- **事件驱动**：业务侧上报任意结构化事件，任务系统自动匹配并更新进度
- **字段过滤器**：任务可设过滤条件（如「击杀 bossId=dragon 的 Boss 才算」），运营在管理台配置，无需改代码
- **多级阶梯奖励**：一个任务可有多个阈值，各档独立领取
- **重置周期**：每日 / 每周 / 永不重置，按任务类型分别配置
- **分类管理**：日常、周常、成就、活动等分类，方便前端分组展示
- **活动积分联动**：任务完成可自动转发为活动积分

## 适合谁

| 角色 | 关注点 |
|------|--------|
| 决策者 / 老板 | 任务系统是驱动玩家行为、延长游戏时长的核心机制 |
| 产品经理 | 灵活配置各类任务，不同活动可快速设计不同的任务体系 |
| 运营人员 | 管理台配置任务和奖励，查看完成率，优化任务设计 |
| 研发团队 | 接入事件上报接口，之后所有任务逻辑通过配置管理 |
