ApolloKit 把「业务动作 → 事件」抽象成一套统一的事件中心。每个模块在关键操作(签到成功、兑换完成、邮件已读、抽奖中奖……)后都会产出一条规范化事件,既写进内部存档,也可以经 Webhook 推给你的下游系统。

## 事件模型

事件分两类:

- **internal(内建)**:ApolloKit 模块自身产生的事件,schema 由 ApolloKit 维护,向下兼容。
- **external(外部)**:你通过事件中心上报的自定义事件,schema 由你维护。可以从 external 升级为 canonical(接管命名 + 字段固化),避免后来的事件定义漂移。

事件名一律走点分命名空间,例如:

```
check_in.checked_in
exchange.completed
item.granted
mail.claimed
task.progress
lottery.drawn
```

完整事件清单随 `apps/server/src/modules/*/service.ts` 中 `logger.record(...)` 的调用演进。管理台「事件中心」页面展示当前 org 所有已上报过的事件 + schema 推断结果。

## 查看当前 org 的事件目录

管理端 API:

```http
GET /api/event-catalog
x-api-key: <admin-api-key>
```

返回合并视图(internal + external),每条包含事件名、当前状态(inferred / canonical)、字段 schema、最近一次出现时间。

```http
GET /api/event-catalog/{name}
PATCH /api/event-catalog/{name}   # 仅 external 可编辑,会自动升级为 canonical
```

## 订阅 Webhook

> 当前 MVP 阶段,Webhook 订阅 + 重试投递能力在活动模块已落地(`/api/activity/webhooks/*`),通用化到全局事件中心的订阅列表仍在路上。本页描述的是**契约**——订阅开通后你会拿到的 payload 形状。

订阅流程:

1. 在管理台或 API 侧为某个事件名(或前缀匹配)登记一个 endpoint(你能接收 HTTPS POST 的 URL)。
2. ApolloKit 为该订阅签发一份 webhook secret。
3. 发生事件时,ApolloKit 异步投递 JSON 到你的 endpoint,header 里带签名。

## Payload 形态

```json
{
  "id": "evt_01hx...",
  "name": "check_in.checked_in",
  "organizationId": "org_...",
  "endUserId": "player_42",
  "occurredAt": "2026-04-21T03:14:15.123Z",
  "data": {
    "configId": "...",
    "configAlias": "daily",
    "currentStreak": 3,
    "justCompleted": false
  },
  "delivery": {
    "attempt": 1,
    "webhookId": "wh_..."
  }
}
```

字段约定:

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 事件唯一 id,**幂等键**——同一个 id 可能因重试多次投递。 |
| `name` | string | 事件名。 |
| `organizationId` | string | 所属项目 id。 |
| `endUserId` | string \| null | 触发事件的终端用户(无则为系统事件)。 |
| `occurredAt` | ISO8601 | 事件实际发生的时间,不是投递时间。 |
| `data` | object | 事件特定字段。形状见 `/api/event-catalog/{name}` 的 schema。 |
| `delivery.attempt` | number | 第几次投递(从 1 开始)。 |
| `delivery.webhookId` | string | 订阅的 id,用来对账。 |

## 签名校验

每次投递带两个 header:

```
x-apollokit-signature: sha256=<hmac-hex>
x-apollokit-timestamp: <unix-seconds>
```

签名算法(推荐实现):

```ts
const payload = `${timestamp}.${rawBodyText}`;
const expected = crypto.createHmac('sha256', webhookSecret)
  .update(payload)
  .digest('hex');
// 常量时间比较
```

校验时:

1. 比较 `sha256=...` 与你算出来的 hex。
2. 验证 `timestamp` 与服务器时间差小于 5 分钟(抵御重放)。
3. 通过后再处理 `data`。

## 幂等处理

因为重试存在,同一个 `id` 可能被你收到 2+ 次。接收端务必做去重:

```ts
const seen = await kv.get(`webhook:${event.id}`);
if (seen) return new Response('', { status: 200 });
await kv.put(`webhook:${event.id}`, '1', { expirationTtl: 24 * 3600 });
// ...处理逻辑...
```

## 重试与退避

当前契约:2xx 算投递成功;3xx/4xx/5xx 或超时(5 秒)算失败。失败按指数退避重试,最多 8 次,跨 24 小时。超过次数的事件会标记 `dead_letter`,在管理台可查。

## 本地联调

可以用 `webhook.site` 或 `smee.io` 拿一个临时 endpoint 订阅,看到真实 payload 后再写签名校验。上生产前务必打开签名校验并关掉 dev 订阅。
